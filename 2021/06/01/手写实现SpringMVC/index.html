1. Spring 与 SpringMVC 的区别

Spring 框架：两大特性 IOC 和 AOP 。

  • SpringIOC 特性：让 Spring 可以理解为一个管理 Bean 的容器
  • SpringMVC 特性：提供了面向切面编程，可以在不改变源代码的基础上，将模块通用的功能提取出来，不仅降低了代码的冗余，还大大降低了模块间的耦合性

SpringMVC 框架：核心类是 DispatcherServlet ，其实 SpringMVC 的本质就是一个 Servlet ，用来拦截前台的请求，调用 Service 层和 持久层 ，返回数据再通过 SpringMVC 把数据渲染成视图并返回给前台。而且 SpringMVC 要基于 Spring 框架才能跑起来。

2. SpringMVC 的执行流程

2.1流程图

SpringMVC的web框架围绕DispatcherServlet设计，DispatcherServlet的作用是将请求分发到不同的处理器，同其它的框架一样，SpringMVC以请求为驱动，围绕一个中心Servelet分派请求并提供相应的功能，DispatcherServlet是一个实际的Servelet。

2.2组件作用

  • DispatcherServlet：前端控制器。

    它就相当于 mvc 模式中的 c， dispatcherServlet 是整个流程控制的中心，由它调用其它组件处理用户的请求， dispatcherServlet 的存在降低了组件之间的耦合性。

    作用：DispatcherServlet负责拦截处理用户发起的请求。

  • HandlerMapping：处理器映射器

    作用：HandlerMapping 负责根据用户请求的url找到 Handler 即处理器。

  • HandlerExecution：具体的Handler

    HandlerExecution 负责根据用户请求的url找到具体的控制器Controller。

  • Handler：处理器 (自己写的Controller类中的方法)

    它就是我们开发中要编写的具体业务控制器。由 DispatcherServlet 把用户请求转发到 Handler。

  • HandlAdapter：处理器适配器

    作用1：按照特定的规则去执行Handler；

    作用2：将Controller返回的ModelAndView回传给DispatcherServlet。

    注：Model表示具体的数据，View表示视图

  • View Resolver：视图解析器

    作用：View Resolver 负责将处理结果生成 View 视图， View Resolver 首先根据逻辑视图名解析成具体的页面地址，再生成 View 视图对象，最后对 View 进行渲染将处理结果通过页面展示给用户。

  • View：视图

    作用：向用户展示结果

2.3执行流程

  1. 用户在浏览器中输入url，被dispatcherServlet 拦截；
  2. dispatcherServlet 调用HandlerMapping找到相应的Handler；
  3. HandlerExecution就是具体的Controller；
  4. HandlerExecution将解析的Controller回传给dispatcherServlet ；
  5. dispatcherServlet 调用HandlAdapter去适配相应的Controller；
  6. HandlAdapter找到相应的Controller去执行；
  7. Controller将执行结果返回给HandlAdapter；
  8. HandlAdapter将执行数据回传给dispatcherServlet ，如ModelAndView；
  9. dispatcherServlet调用View Resolver得到相应的页面地址；
  10. View Resolver解析得到相应的页面地址，回传给dispatcherServlet；
  11. dispatcherServlet根据View Resolver的结果调用相应的视图。

3.3 package-springframework

3.3.1 package-web

增加一个 web 包，用来放我们的核心类 DispatcherServlet ，表示与 web 处理应用有关。

DispatcherServlet 类：重写了父类 HttpServlet 中三个重要的方法：init 方法、doGet 方法、doPost 方法

**init 方法：**初始化了一个 Spring 容器，调用方法完成：读取配置文件、扫描包下所有的 Controller、实例化所有 Controller 、并绑定 URL 路由

doGet 方法：初始化之后，当一个请求到来，首先到达 doGet 和 doPost 方法。

doPost 方法：略

**doDispatch 方法：**我们自定义的实际处理请求的方法，主要逻辑就是从请求中获取请求的 Url 也就是请求的方法，然后用一个数组存放请求的参数，最后方法和方法的参数都获取到之后，通过反射的方式调用方法处理参数。

doLoadConfig 方法：读取配置文件，获取 web.xml 文件中 DispatcherServlet 的配置文件路径

doScanner 方法：扫描包下所有的 Controller，获取要扫描包的范围，并将范围内所有类放入 classNames 集合中

doInstance 方法：实例化所有 Controller ，遍历 classNames 集合中所有的类，如果是加了 @Controller 注解的类就将 BeanDefinition 先注入到容器中，那么容器中有 Bean 是没有被实例化的，再调用 refreshBeanFactory 方法手动刷新，也就是将 BeanDefinition 中的 Bean 属性进行注入就将所有的 Controller 进行实例化了。

initHandlerMapping 方法：遍历类集合 classes 如果类上有 RequestMapping 注解，根据注解的值来拼接 URL，并将 URL 与类（Controller）的映射存储在 controllerMap 中，将 URL 与 方法 的映射存储在 handlerMapping 这个 map 中
